Combiner

  • We specify a combiner which performs local aggregation on the map results after being sorted on keys

Reducer

  • Interface: public class Reducer<KEYIN, VALUEIN, KEYOUT, VALUEOUT>

  • Reducer has three phases:

    1. Shuffle

      • The Reducer copies the sorted output from each Mapper using HTTP across the network

    2. Sort

      • The framework merge sorts Reducer inputs by keys (since different Mappers may have output the same key)
      • Secondary sort on intermediate keys is allowed
      • The shuffle and sort phases occur simultaneously

    3. Reduce

      • Implemented in reduce() method

Job Run

  • Entities are involved in a MapReduce job run:
    1. Client, which submits the MapReduce job
    2. YARN resource manager, which coordinates allocation of cluster resources
    3. YARN node managers, which launch and monitor compute containers
    4. MapReduce application master, which coordinates the tasks of the MapReduce job
    5. Distributed filesystem, which stores and shares job files between the other entities

  • Job Submission

    • Step 2. The client asks the resource manager for a new job ID
    • Step 3. The client copies resources to the filesystem
    • Step 4. The client tells the resource manager the job is ready for execution

  • Job Initialization

    • Step 5. The resource manager hands off requests to YARN scheduler, which allocates a container to run the application (managed by Node Manager)
    • Step 6. The application master keeps track of job progress
    • Step 7. The application master retrieves input splits from the file system
    • The application master may run tasks of a “small” job within the same JVM as itself, without asking for new containers

  • Task Assignment

    • Step 8. For large jobs, the application master requests containers for all map and reduce tasks in the job from the resource manager

  • Task Execution

    • Step 9. The application master starts the container by contacting the node manager, under a YarnChild class
    • Step 10. The YarnChild class localizes resources (e.g., job configuration and JAR file)
    • Step 11. Run the map/reduce task

Handling Failures

  • Task failure
    • If a task is crashed or hanging, it fails
    • The application master reschedules the task on a different node manager. If any task fails 4 times (default), whole job fails
  • Application master failure
    • The resource manager detects the failure and starts a new master instance in a new container, and tries to recover the state of tasks using job history
  • Node manager failure
    • The resource manager recovers task/application master failures as above, and removes the failed node manager
  • Resource manager failure
    • Single point of failure
    • Need active and standby resource managers

Additional

Configuration Tuning

  • Specify configurations in XML file with property-value pair

Optimization

  • Speculative execution

    • Job execution time is bottlenecked by the slowest running task
    • Straggler: a machine that takes unusually long time to finish the last few tasks (it is hard to determine it is failure or not as it is still running)
    • If a task is running slow, Hadoop launches another equivalently, task as a backup (When the task finishes, any duplicate tasks are killed)

  • Skipping bad records

    • Bad records throw runtime exceptions, causing a task to retry or even halt (after 4 retries)
    • If skipping mode is enabled, failed records are skipped (some small parts is not that importance, ignore them may not affect the big picture, which is acceptable)

Counters

  • a useful channel for gathering statistics about a job, Hadoop maintains built-in counters for a job
  • User-defined counters:
    • Java enum type
    • Counters are global: MapReduce aggregates them across all map and reduce tasks

MapReduce Algorithm Design

Local Aggregation

  • In Hadoop, intermediate results (i.e., map outputs) are written to local disk before being sent over the network
  • Local aggregation of intermediate results reduces the number of key-value pairs that need to be shuffled from the mappers to the reducers

  • Default combiner: Aggregate map outputs with the same key, acts like a mini-reducer

In-mapper Combiners

  • Version 1:
    • Emits a key-value pair for each unique term per document

  • Version 2:

    • Recall a map object is created for each map task
    • Aggregate all data appearing in the input block processed by the map task

  • Advantages:

    1. Provide control over where and how local aggregation takes place.
    2. In-mapper combiners are applied inside the code. Default combiners are applied to the map outputs.

  • Disadvantages:

    1. States are preserved within mappers → potentially large memory overhead.
    2. Potential order-dependent bugs.

Combiner Design

  • Combiner and reducer must share the same signature

    • Combiner is treated as mini-reducer
    • Combiner input and output key-value types must match the reducer input key-value type

  • Combiner are optional optimizations

    • With/without combiner should not affect algorithm correctness

Pairs and Stripes

  • constructing complex keys and values improves the performance of computation

Co-occurrence Matrix

  • Example: building a word co-occurrence matrix over a text collection

    • $M = n \times n$ matrix (n = number of unique words), $m_{ij} = $ number of times word $w_i$ co-occurs with word $w_j$ within a specific context

    • Challenge: Space requirement: $O(n^2)$

    • Approach: Mappers generate partial counts, Reducers aggregate partial counts

  • Pairs

    • Mapper:
      • Emits intermediate key-value pairs with each co-occurring word pair and integer 1

    • Reducer:

      • Sums up all values associated with the same co-occurring word pair
      • MapReduce execution framework guarantees that all values associated with the same key are brought together in the reduce

    • Advantages:

      1. Easy to understand and implement
    • Disadvantages:
      1. Generate many key-value pairs

  • Stripes

    • Mapper:
      • For each particular word, stores co-occurrence information in an associative array
      • Emits intermediate key-value pairs with words as keys and corresponding associative arrays as values

    • Reducer:

      • Sums all the counts in the associative arrays
      • MapReduce execution framework guarantees that all associative arrays with the same key are brought together in the reducer


    • Advantages:

      1. Generate fewer key-value pairs
      2. Make better use of combiners
    • Disadvantages:
      1. Memory size of associative arrays in mappers could be huge

  • Stripes much faster than pairs

Relative Frequencies

  • Stripes Approach
    Similar to the approach of constructing co-occurrence matrix: Sum all the counts in the associative array for each word

  • Pair Approach

    • Mapper emits (a,*) for every word being observed
    • Mapper makes sure same word goes to the same reducer (use partitioner)
    • Mapper makes sure (a,*) comes first, before individual counts
    • Reducer holds state to remember the count of (a,*), until all pairs with the word “a” have been computed

Order Inversion

  • Computing relative frequencies requires marginal counts because it cannot be computed until getting all counts
    For every pair, the mapper adds the count to the marginal count
    When all pairs are processed, the mapper also emits the marginal count
    Using order inversion, the marginal count arrives before the pairs at the reducer
    The reducer remembers the marginal count, and computes the relative frequency for each pair being observed
  • Control the sort order of the intermediate key so that the marginal count comes before individual counts job.setSortComparatorClass()

Secondary Sort

MapReduce sorts input to reducers by key, but values may be arbitrarily ordered. What if want to sort value also?
E.g., Sensors record temperature over time (id, time t, temperature v)

  • Value-to-key conversion
    Each mapper emits (id, t) → v. Let execution framework do the sorting.
    Preserve state across multiple key-value pairs to handle processing.

Tools for Synchronization

  • Cleverly-constructed data structures
    Bring data together
  • Sort order of intermediate keys
    Control order in which reducers process keys
  • Partitioner
    Control which reducer processes which keys
  • Preserving state in mappers and reducers
    Capture dependencies across multiple keys and values
